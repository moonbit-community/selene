## Description
Define the MoonBit-to-JavaScript boundary for `selene-webgpu` before moving implementation logic. The boundary should expose browser/WebGPU capabilities as direct syscalls or opaque handle operations, not high-level renderer behavior.

## Design
Introduce or outline a `webgpu_sys` style layer with functions for device/context setup, resource creation/release, buffer writes, texture/sampler/pipeline/bind-group creation, encoder/pass operations, draw calls, submit, and browser-only DOM/audio/input calls. Avoid functions that decide batching, mesh generation, lighting policy, or render ordering.

### Boundary Rule
MoonBit owns renderer policy; JavaScript owns direct browser/WebGPU calls only.

### Allowed Syscall Groups
Device/context:
- `sys_request_device`: JS calls `navigator.gpu`, `requestAdapter`, and `requestDevice`; MoonBit owns backend startup state.
- `sys_configure_canvas`: JS configures `GPUCanvasContext`; MoonBit owns viewport and surface configuration.

Resources:
- `sys_create_buffer`: JS creates a `GPUBuffer` and returns an opaque handle; MoonBit owns buffer purpose, cache key, and lifetime.
- `sys_write_buffer`: JS calls `GPUQueue.writeBuffer`; MoonBit owns payload contents and write timing.
- `sys_destroy_resource`: JS destroys/deletes opaque browser objects; MoonBit owns release policy and ordering.
- `sys_create_texture`: JS creates/uploads browser texture data; MoonBit owns asset identity and texture descriptors.
- `sys_create_sampler`: JS creates `GPUSampler`; MoonBit owns sampler keys.
- `sys_create_bind_group`: JS creates `GPUBindGroup` from opaque handles; MoonBit owns layout/resource keys and reuse policy.
- `sys_create_render_pipeline`: JS creates `GPURenderPipeline`; MoonBit owns pipeline keys and shader variant selection.

Encoder/pass:
- `sys_begin_encoder`: JS creates `GPUCommandEncoder`; MoonBit owns frame order and pass graph.
- `sys_submit_encoder`: JS finishes/submits command buffers; MoonBit owns submit ordering and retired-resource fence policy.
- `sys_begin_render_pass`: JS begins `GPURenderPassEncoder`; MoonBit owns pass target, viewport, clear/load policy.
- `sys_set_pipeline`: JS binds an opaque pipeline handle; MoonBit chooses the pipeline.
- `sys_set_bind_group`: JS binds an opaque bind group handle; MoonBit chooses the bind group.
- `sys_set_vertex_buffer`: JS binds an opaque buffer handle; MoonBit owns slot, offset, and range.
- `sys_draw`: JS calls `draw`; MoonBit owns draw counts and instance counts.
- `sys_end_render_pass`: JS ends the pass; MoonBit owns pass lifetime.

Browser platform:
- `sys_measure_text`: JS queries browser text metrics; MoonBit owns text layout requests.
- `sys_browser_audio`: JS calls browser audio APIs; MoonBit owns playback state.
- `sys_pointer_dom`: JS reads DOM pointer and lock state; MoonBit owns normalized input state.

### High-Level JS Entrypoints To Retire
- `webgpu_draw_cube_3d`
- `webgpu_draw_sphere_3d`
- `webgpu_draw_cylinder_3d`
- `webgpu_draw_colored_triangles_3d`
- `webgpu_draw_textured_triangles_3d`
- `webgpu_set_lighting_3d`
- `webgpu_set_lighting_mode_3d`
- JS-side primitive mesh builders and batch/retained-lighting decisions inside `canvas.mbt`

### Non-Goals
- Do not rewrite the renderer in this issue.
- Do not move renderer policy into standalone `.js` files.
- Do not expose these syscall names as public user API until the implementation slices need them.

## Acceptance Criteria
- [x] A concrete syscall boundary is represented in code or an issue-local implementation note with function groups and ownership rules.
- [x] The boundary distinguishes opaque GPU handles from MoonBit-owned render/resource keys.
- [x] Existing high-level JS entrypoints that should be retired are identified.
- [x] No renderer policy is added to JavaScript while defining this boundary.
