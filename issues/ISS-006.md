# ISS-006: Move WebGPU backend ownership back to MoonBit

## Metadata

- Status: open

- Created: 2026-06-01
- Updated: 2026-06-01
- External ref: none

## Description
The WebGPU backend currently keeps too much renderer logic inside embedded JavaScript strings. Since MoonBit compiles to JavaScript, WebGPU should not be implemented as a second hand-written JS renderer. The intended ownership is MoonBit-first: renderer policy, batching, resource decisions, mesh generation, lighting packets, telemetry, and tests belong in MoonBit. JavaScript should remain a thin FFI/syscall layer over browser and WebGPU APIs.

## Design
Treat this as an architecture reset for `selene-webgpu`. Keep JS only where MoonBit cannot directly express browser/WebGPU operations: direct DOM access, `navigator.gpu`, `GPUDevice` object operations, command encoder/pass calls, audio, and pointer APIs. Move all non-trivial decisions out of `canvas.mbt` embedded JS in small verifiable slices.

## Acceptance Criteria
- [ ] `canvas.mbt` no longer owns renderer policy such as primitive batching, retained batch decisions, mesh generation, lighting filtering, or telemetry.
- [ ] WebGPU FFI functions are thin direct bindings to browser/WebGPU capabilities or opaque handle operations.
- [ ] MoonBit owns WebGPU resource keys, render command IR, primitive mesh generation, batch formation, retained/static policy, lighting packet construction, and telemetry.
- [ ] Existing 2D, 3D primitive, texture, shadow, audio, input, and editor-preview behavior remains covered by tests or documented manual smoke checks.
- [ ] `moon check --target all --deny-warn` passes when all child issues are closed.
