# ISS-005: Add retained static batches and cheaper WebGPU lighting options

## Description
After primitive mesh caching and instancing, add higher-level performance options for large mostly-static 3D scenes. Tile-heavy scenes should be able to avoid rebuilding static world submissions every frame, and low-cost web lighting modes should be available when forward lighting over all active lights is too expensive.

## Design
Introduce an internal or public retained static batch path only after the lower-level WebGPU renderer is stable. Consider explicit static/dynamic submission separation, retained mesh buffers for static world geometry, and per-object or per-batch light filtering. Do not hide visual changes behind the default path; cheaper lighting modes should be explicit.

## Acceptance Criteria
- [ ] Static 3D world geometry can use a retained path that avoids per-frame command reconstruction and vertex upload.
- [ ] Dynamic 3D objects continue to render correctly alongside retained static batches.
- [ ] WebGPU 3D lighting has an explicit lower-cost option or light filtering path with documented behavior in code comments.
- [ ] A primitive-heavy static scene shows reduced per-frame upload/command telemetry compared with the immediate path.
- [ ] `moon test selene-webgpu/src --target js --deny-warn` passes.
